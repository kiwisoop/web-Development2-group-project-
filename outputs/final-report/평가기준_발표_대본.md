# Sport Analysis Dashboard 평가기준 맞춤 발표 대본

## 1. 표지
안녕하세요. 저희 팀은 축구, 야구, e스포츠 경기 데이터를 하나의 흐름으로 통합하고, 경기 상세 기록과 AI 분석을 함께 제공하는 Sport Analysis Dashboard를 개발했습니다. 이번 발표는 기능 나열보다 평가기준에 맞춰 문제 정의, 설계 판단, AI 협업 과정, 검증 결과를 중심으로 설명하겠습니다.

## 2. 평가 기준에 맞춘 발표 구조
평가기준은 산출물 30%, 과정 45%, 성찰과 발표 25%로 나뉩니다. 그래서 저희 발표도 완성 화면만 보여주는 방식이 아니라, 왜 이런 구조를 선택했는지, AI를 어떻게 검증했는지, 어떤 오류를 해결했는지를 함께 보여주는 흐름으로 구성했습니다.

## 3. 문제 정의
스포츠 팬은 종목별로 다른 사이트를 오가며 일정, 결과, 순위, 선수 기록을 확인해야 합니다. MLB, K리그, LCK는 데이터 구조와 경기 상태 기준이 서로 다르기 때문에, 단순히 데이터를 모으는 것보다 같은 사용자 흐름으로 정리하는 것이 핵심 문제였습니다.

## 4. 아키텍처 결정
프론트엔드는 React SPA, 백엔드는 Spring Boot REST API로 분리했습니다. 종목별 API 구조가 다르기 때문에 백엔드는 축구, 야구, e스포츠 패키지를 분리했습니다. 대신 사용자는 홈, 경기센터, 순위, 스포츠 허브라는 공통 화면 기준으로 탐색하도록 설계했습니다.

## 5. 주요 구현 범위
구현한 핵심 기능은 홈, 경기센터, 경기 상세, 스포츠 허브, 순위, 관리자입니다. 홈에서는 실제 LIVE 경기와 3일 내 예정 경기만 보여주고, 경기센터에서는 날짜·종목·상태 필터로 목록을 조회합니다. 상세 화면에서는 기록, 팬 투표, 채팅, AI 분석까지 이어집니다.

## 6. 종목별 차별화
축구는 K리그 일정과 순위, 상세 분석을 연결했습니다. 야구는 MLB Stats API 기반으로 오늘 경기, 라인스코어, 선발 투수, 핵심 타자, AI 리포트를 제공합니다. e스포츠는 Cito API와 DB 선수 지표를 결합해 LCK 일정, 경기 결과, 팀 정보, 게임별 통계를 보여줍니다.

## 7. AI 활용 전략
초기에는 단순히 기능 구현을 요청했지만, 후반에는 목표, 제한조건, 기존 파일, 검증 명령을 함께 제공했습니다. 예를 들어 기존 기능 제거 금지, 실제 LIVE 기준, Spring Security 미도입 같은 조건을 반복해서 AI가 프로젝트 방향을 벗어나지 않도록 했습니다.

## 8. 코드 검증과 비판적 사고
AI가 만든 코드를 그대로 쓰지 않았습니다. 관리자 빈 화면, Maven wrapper 실패, 경기센터 필터 오류, 팀 로고 누락 같은 문제를 실제 실행 결과로 확인했습니다. 특히 Maven wrapper 문제는 코드 오류가 아니라 Windows용 스크립트 처리 문제였고, 수정 후 백엔드 컴파일을 통과시켰습니다.

## 9. 최종 검증 결과
최종 검증으로 백엔드 컴파일, 프론트엔드 린트, 프론트엔드 빌드를 모두 통과했습니다. 검증 명령은 보고서와 README에도 남겨 발표 직전 재확인이 가능하게 했습니다. 외부 API 키와 Oracle DB 접속은 발표 환경에서 다시 확인해야 하는 현실적 제한으로 기록했습니다.

## 10. 윤리와 학술적 정직성
AI 사용 과정에서 API 키와 개인정보를 AI에게 직접 입력하지 않았고, 프롬프트 로그를 실제 흐름에 맞게 남겼습니다. AI 분석은 승부 예측이 아니라 경기 이해를 돕는 보조 리포트로 안내했습니다. 개인 성찰은 평가기준상 AI 작성 금지이므로 각 팀원이 직접 작성해야 합니다.

## 11. 팀워크와 역할 분담
종목별로 축구, 야구, e스포츠 담당을 나누고, 인증·경기센터·관리자·UI 통일 같은 공통 기능은 함께 조정했습니다. 특히 화면마다 기준이 달라지는 팀 로고, 실시간 상태, 순위 기준은 팀 피드백을 바탕으로 통합했습니다.

## 12. 시연 흐름
시연은 홈, 경기센터, 경기 상세, 스포츠 허브, 관리자 순서로 진행하는 것이 좋습니다. 홈에서는 실시간 기준, 경기센터에서는 필터 기준, 상세에서는 MLB 기록과 팬 투표, 스포츠 허브에서는 종목별 화면, 관리자에서는 데이터 동기화와 권한 제한을 보여줍니다.

## 13. 예상 질문 대응
예상 질문은 네 가지입니다. 첫째, 왜 AI 코드를 그대로 쓰지 않았는가. 둘째, 왜 HttpSession을 선택했는가. 셋째, AI 분석이 예측인지 요약인지. 넷째, 외부 API 실패 시 어떻게 대응하는가입니다. 답변의 핵심은 현실적 제한과 검증 과정을 설명하는 것입니다.

## 14. 마무리
이 프로젝트의 핵심은 종목별 데이터를 단순히 모은 것이 아니라 하나의 탐색 흐름으로 통합한 것입니다. 또한 AI를 코드 생성 도구로만 사용하지 않고, 검증 가능한 협업 도구로 사용한 과정 자체를 평가 증거로 남겼습니다.
